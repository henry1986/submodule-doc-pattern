ドキュメント設計

Git Submodule Docs Pattern

Gitサブモジュールを介してライブラリドキュメントを分離し、バージョン管理するための標準化されたパターン。開発者と自律型AIエージェント向けに最適化されています。

ドキュメントの分離

ドキュメントとコードは別々のリポジトリに置かれます。これにより、仕様書のみを必要とする利用者がクローンする際の容量と帯域を節約できます。

バージョンの同期

サブモジュールは特定のGitタグ（例：v1.0.0）にチェックアウトされます。これにより、ドキュメントは常に使用中のコードバージョンと100%一致します。

AIエージェント対応

ワークスペース内のAIエージェントは、Webやソースコードを解析することなく、標準化されたパス docs/lib-docs/ を介してAPIマニュアルに即座にアクセスできます。

核心的な仕組みの詳細

my-library のようなライブラリを作成する場合、パッケージマネージャーに公開します（例：Maven/Gradleでローカルの mavenLocal() にインポート）。並行して、ドキュメント用のリポジトリ（例：my-library-docs）を作成します。

ライブラリプロジェクトでは、このドキュメントリポジトリは docs/releases に書き込み権限付きで統合されます。ライブラリを使用するコンシューマープロジェクトでは、同じドキュメントリポジトリが 読み取り専用のサブモジュール として docs/lib-docs/my-library に統合され、対応するリリースバージョンタグに固定されます。 外部依存関係用の docs/lib-docs/ に加えて、プロジェクトフォルダ docs/ には、内部プロジェクトドキュメント用の他のディレクトリ（docs/design/ や docs/internal/ など）も当然含めることができます。

エージェントへのメリット：

コンシューマープロジェクトの編集を依頼されたAIエージェントは、標準化された docs/lib-docs/ フォルダを検出し、使用されているバージョンのMarkdownマニュアルを直接読み込み、推測に頼らずに正しいAPI呼び出しコードを生成します。

1. ドキュメントリポジトリの作成

まず、マニュアル、統合例、API仕様のみを管理する専用のGitHubリポジトリ（例：my-library-docs）を作成します。ソースコード自身はこの存在を知りません。

Gitコマンド生成器

プロジェクトの詳細を入力すると、対応するGitコマンドが即座に生成されます。

フェーズ A：ライブラリリポジトリでの設定 
# 1. GitHub CLI を使用して新しいドキュメントリポジトリを作成
gh repo create my-organization/my-library-docs --public --clone

# 2. メインのライブラリリポジトリにサブモジュールとして追加（書き込み権限）
git submodule add git@github.com:my-organization/my-library-docs.git docs/releases
フェーズ B：コンシューマーリポジトリでの統合 
# 1. 読み取り権限でドキュメントサブモジュールを追加
git submodule add git@github.com:my-organization/my-library-docs.git docs/lib-docs/my-library

# 2. 特定のバージョンタグを取得して固定
cd docs/lib-docs/my-library
git fetch --tags
git checkout tags/v1.0.0
cd ../../

# 3. 固定したサブモジュールをコミット
git add docs/lib-docs/my-library
git commit -m "chore: pin my-library docs to v1.0.0"
